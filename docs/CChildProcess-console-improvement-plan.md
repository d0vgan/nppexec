## 1. Goals

| Scope | Description |
|-------|-------------|
| **A — Reliability** | Output not stuck in pipe buffers; timely display for line-oriented tools (`cl`, `gcc`, `git`, unbuffered Python, `\r` progress). |
| **B — ANSI-ish console** | Colors, cursor motion, erase line, basic spinners (within RichEdit limits). |
| **C — Full TUI** | `vim`, `less`, curses — may require embedded terminal or external window; likely out of scope for RichEdit alone. |

Recommended release order: **A → B → C**.

### 1.1 Architectural principles (generic console only)

NppExec is building a **general** console host: the same pipe/ConPTY transport, read loop, VT rendering, and input path must work for **arbitrary** child programs without special cases.

| Rule | Requirement |
|------|-------------|
| **No application-specific logic** | Do **not** branch on executable name/path (`cmd`, `python`, `gawk`, `npm`, etc.), file extension, or “known interactive” programs. |
| **No per-app `CreatePseudoConsole` policy** | Do **not** choose ConPTY vs pipes, flags, or geometry per application. Use global options (`npe_console u+` / `u-`, encoding, geometry from Console UI). |
| **No per-app kill/input/output hacks** | Avoid per-exe kill/input/output hacks. Use one I/O model and one kill policy for all children **except the explicitly allowed `cmd.exe` Ctrl-Break exception** (see §1.1.1). |
| **Wider fix when behavior differs** | If one program misbehaves, fix the **generic** layer (VT erase, echo policy, buffering docs, flush rules)—not a targeted exception for that program. |
| **Suspicious legacy code** | Treat existing app-specific branches as **unwanted**; remove when touched (see §1.1.1), do not extend them. |

**Acceptable exceptions (not “app logic”):**

- User/script **options** (`npe_console`, encoding, PseudoConsole on/off) — global policy, applies to the next spawned child only.
- **OS/API** requirements (e.g. `STARTF_USESTDHANDLES` only in pipe mode) — transport-level, not executable identity.
- **Documentation / repro matrix** naming `cmd`, `python`, `gawk` as *test cases*, not as code branches.

#### 1.1.1 Legacy application-specific code (remove, do not extend)

| Location | What | Action |
|----------|------|--------|
| `ChildProcess.cpp` (`WaitForExit` kill path) | Skips Ctrl-Break/Ctrl-C for `cmd` / `cmd.exe` only | **Allowed** — keep (cmd is a shell hosting another process; Ctrl-Break is delegated and not a reliable “close shell” signal). Keep this exception bounded to kill-signal choice only. |
| Plan Phase 3 step 6 (removed) | “Auto-select PseudoCon vs pipes by executable type” | **Do not implement** — violates generic transport |
| Comments implying “python -i needs X” in hot path | e.g. reader wake comments | Keep as repro notes only; behavior must stay generic |

**P3.5 scope (reframed):** review **global** `CreatePseudoConsole` API flags (e.g. `PSEUDOCONSOLE_INHERIT_CURSOR`) and document the chosen default (`flags = 0` today). **Not** “flags for specific apps.”

## 2. Current architecture

### 2.1 End-to-end flow

```
[Main thread]
  ConsoleDlg RichEdit  ←→  CNppExecCommandExecutor (input)
       ↑                           │
       │ PrintOutput               │ WriteChildProcessInput → CChildProcess::WriteInput
       │                           ↓
[Background script thread]
  CScriptEngine::Do → CChildProcess::Create (blocks until child exits)
                         → readPipesAndOutput (poll stdout pipe)
                         → CNppExecConsole::PrintOutput / ProcessSlashR / ProcessSlashB

[Child process]
  stdin/stdout (anonymous pipes OR Windows PseudoConsole / ConPTY)
```

### 2.2 Key code locations

| Component | File | Notes |
|-----------|------|--------|
| Child process lifecycle | `NppExec/src/ChildProcess.cpp`, `ChildProcess.h` | `Create`, `readPipesAndOutput`, `WriteInput` |
| Script entry | `NppExec/src/NppExecScriptEngine.cpp` | `Do()` — `Create()` does not return until child exits |
| Console input routing | `NppExec/src/NppExecCommandExecutor.cpp` | `ExecuteChildProcessCommand`, `WriteChildProcessInput*` |
| RichEdit UI | `NppExec/src/DlgConsole.cpp` | Enter, Ctrl+Z, etc. → executor |
| Console output | `NppExec/src/NppExec.cpp` | `CNppExecConsole::PrintOutput`, `ProcessSlashR`, `ProcessSlashB` |
| RichEdit wrapper | `NppExec/src/CAnyRichEdit.cpp` | `AddLine` / `AddStr`; EOL is `\r` (`base.h`) |

### 2.3 Bidirectional console (already implemented)

While a child is running, the **same RichEdit** is used for **output and input**:

| Direction | Path |
|-----------|------|
| **Output** | Child → stdout pipe → `readPipesAndOutput` → `PrintOutput` → RichEdit |
| **Input** | RichEdit (user at bottom) → `ConsoleDlg` → `ExecuteChildProcessCommand` → `WriteChildProcessInput` → **`CChildProcess::WriteInput`** → stdin pipe |

`WriteInput` encodes Unicode to OEM/UTF-8/ACP and writes to `m_hStdInWritePipe`. Enter sends **line text** then **newline separately** (`WriteChildProcessInput` + `WriteChildProcessInputNewLine` using `GetNewLine()` — `\r` under PseudoConsole, else user “Enter” setting).

**The gap is not missing stdin.** The gaps are:

- **Output:** pipe polling (`PeekNamedPipe`), line-oriented parsing, ANSI stripped not rendered.
- **Input:** line-oriented + optional flush; not raw terminal keys (arrows, Tab, etc.) unless added later.

### 2.4 Process creation

**Standard mode** (PseudoConsole off):

- `CreatePipe` for stdin/stdout; stderr merged into stdout pipe.
- `STARTF_USESTDHANDLES`: `hStdInput` / `hStdOutput` / `hStdError` → pipes.

**PseudoConsole mode** (`OPTB_CHILDP_PSEUDOCONSOLE`, Win10 1809+):

- `CreatePseudoConsole` size from Console RichEdit client area + font metrics (defaults 192×50, caps 40–500 × 10–1000); `ResizePseudoConsole` not used yet (P3.3).
- `CreateProcess` with `PROC_THREAD_ATTRIBUTE_PSEUDOCONSOLE`.
- On success, inherited pipe ends to child are closed; plugin holds read/write ends.

### 2.5 Read loop (stdout) — “stuck in pipe”

`readPipesAndOutput`:

1. `Sleep(10)` per inner iteration.
2. `PeekNamedPipe` — if false/0, often skips `ReadFile`.
3. `ReadFile` up to `CONSOLEPIPE_BUFSIZE` (8000).
4. Line buffer: split on `\n`, lone `\r`, `\b`; filters; `PrintOutput`.

Outer loop: `WaitForSingleObject(process, OPTU_CHILDP_CYCLETIMEOUT_MS)` (default **120 ms**) between calls.

Documented issue ([4.8.3.html](NppExec_Manual/4.8.3.html) §9): `PeekNamedPipe` may report empty while data is buffered; blocking `ReadFile` may wait until child flush or exit.

### 2.6 Rendering (not a terminal emulator)

- `escProcess` = `escRemove` — ANSI sequences **stripped**, not rendered (`RemoveAnsiEscSequencesFromLine`).
- PseudoConsole forces strip when option is `escKeepRaw`.
- `\r` → `ProcessSlashR` (clear current RichEdit line); `\b` → `ProcessSlashB`.
- `AddLine` appends RichEdit EOL (`\r`); model is **log + locked input line**, not a character grid.

### 2.7 Threading (current + target)

| Thread | Role | Must not |
|--------|------|----------|
| **Script thread** | `Create()` wait loop, drain queue, `processOutputBuffer`, `PrintOutput` | Block indefinitely on stdout `ReadFile` while child may still run |
| **Stdout reader thread** | Blocking/async read from pipe → enqueue chunks | Call `PrintOutput` or touch RichEdit / console filters |
| **UI / main** | User input, console close, Ctrl+C → `MustBreak` | Assume `Create()` has returned |

`CChildProcess::Create` runs on the **background script thread** and blocks until the child lifecycle completes. Console updates use `PrintOutput` on that same thread (thread-id stack in `CNppExecConsole`).

**Target (Phase 1):** reader thread owns pipe reads; script thread only drains the queue and prints lines.

### 2.8 Implementation invariants (non-negotiable)

Any change to `CChildProcess`, `CChildProcessIo`, or the `Create()` wait/drain/kill path **must preserve these**. Violations caused production deadlocks (gawk, N++ exit blocked) during P1.1–P1.4; treat this list as a **design contract**, not suggestions.

#### Pipe handle lifecycle (MSDN `CreatePipe` pattern)

After `CreateProcess` succeeds, the parent **must close its copies** of handles the child inherited for ends the parent does not use:

| Parent keeps | Parent must close after child start |
|--------------|-------------------------------------|
| `m_hStdOutReadPipe` (read stdout) | `m_hStdOutWritePipe` (child writes stdout/stderr) |
| `m_hStdInWritePipe` (write stdin) | `m_hStdInReadPipe` (child reads stdin) |

Implementation: `CChildProcessIo::onChildProcessStarted()` (all modes, including non–PseudoConsole).

**Why:** If the parent keeps the stdout **write** end open, the read end never sees EOF/`ERROR_BROKEN_PIPE` when the child exits → blocking `ReadFile` waits forever → `stopStdoutReader()` waits forever → **script thread deadlocks** even though `GetExitCodeProcess` shows the child has exited.

#### Blocking I/O placement

| Operation | Allowed thread | Forbidden |
|-----------|----------------|-----------|
| Blocking `ReadFile` on stdout pipe (pipe mode, child alive) | Script thread via **`readPipesDirect`** (legacy HEAD); stdout reader thread **not** started in `Create()` | Reader queue only; `drainStdoutToQueue` while child alive |
| Blocking `WriteFile` / `FlushFileBuffers` on stdin pipe | Stdin **writer** thread only (P1.6) | UI / script thread |
| `FlushFileBuffers` on stdin | Only when `WriteInput(..., bFFlush=true)` (P1.7; default **false**, no in-tree `true`) | Implicit or default flush after every line |
| Stderr (pipe mode) | Merged into stdout write pipe (legacy HEAD); normal text colour | Separate stderr pipe + error colour (P1.8 reverted) |
| Reader: WFMO on pipe handles | **Do not use** — anonymous pipes do not signal data-ready | `WaitForMultipleObjects` (P1.8 regression) |
| Application-specific I/O / kill / transport hacks | **Never** (no executable-name branches, per-app flags, per-app kill lists) | Any `if (app == cmd/python/...)` in `ChildProcess*.cpp`; see §1.1 |
| `PrintOutput` / filters / `processOutputBuffer` | **Script** thread only | Reader thread |
| `PeekNamedPipe` as sole gate before read | Deprecated on hot path; if used, must not skip data that `ReadFile` would return | Replacing drain with peek-only on exit |

#### Exit / break / drain ordering (script thread)

On leaving the `Create()` wait loop, order is **fixed**:

1. If **break** (Ctrl+C, console close, `MustBreak`) and child still running → `stopStdoutReader()` → `stopStdinWriter()` → **kill** (`Kill` / `TerminateProcess`) — **never** drain stdout first.
2. **Only when the child has terminated** (`GetExitCodeProcess != STILL_ACTIVE`, not merely “no peek data”) → `readPipesAndOutput(..., bOutputAll=true)` → `stopStdoutReader()` → `drainStdoutToQueue()` → drain queue → `processOutputBuffer` (flush partial line).
3. Then close handles, print `<<< Process finished` / killed message.

**Never** call `drainStdoutToQueue()` while the child is still `STILL_ACTIVE`. Pipe-mode stdout uses legacy `readPipesDirect` on the script thread (same as git HEAD before P1.4 reader queue).

#### `stopStdoutReader()` must not hang

- Call `CancelSynchronousIo` on the reader thread before join.
- `WaitForSingleObject(reader, timeout)` — **not** `INFINITE` without a cancellation path.
- If the reader does not exit, force termination or close the read pipe as a last resort (document in code comment).

#### Process liveness

The wait loop must treat the child as running only if **`GetExitCodeProcess == STILL_ACTIVE`** (and optionally `WaitForSingleObject` with timeout). Do not infer liveness from pipe peek, queue emptiness, or lack of console output.

#### Buffered child stdout (expectation, not a bug)

Python/gawk **without** unbuffered mode may send **no bytes until exit**. That is **child-side buffering**, not a pipe deadlock. P1 steps must not claim “streaming fixed” when the child never flushes (see B1b / R02b / R08b in baseline).

**Stdin/output coupling:** A program may block on **stdin** (`getline`, `read`, interactive prompt) while **buffered stdout** already contains text the user expects to see (e.g. gawk `print 123` before `getline` — [baseline B1c](CChildProcess-baseline-P0.3.md)). Output may appear only after the user sends a Console line because that is when the runtime flushes or proceeds — not because NppExec failed to read the pipe.

### 2.9 P1 regression lessons (2026-05-25)

| Failure | Cause | Prevention (invariant) |
|---------|--------|-------------------------|
| Ctrl+C / close console ineffective | `readPipesAndOutput(true)` ran **before** kill; script blocked in `drainStdoutToQueue` while gawk alive | §2.8 exit ordering |
| No `<<< Process finished` though gawk exited | Parent held `m_hStdOutWritePipe` → reader/drain block forever | §2.8 close inherited write ends |
| N++ cannot exit | Script thread stuck in `stopStdoutReader(INFINITE)` | §2.8 reader stop timeout + pipe EOF |
| B1b unchanged after P1.3 | Buffered Python — not peek-related | §2.8 buffered expectation; separate ConPTY/P3 track |
| `cmd` — no prompt | Reader-queue path only flushed on `\n`; `readPipesDirect` flushed partial on empty peek | After `popStdoutChunk`, call `processOutputBuffer` with partial flush when `bufLine` non-empty |
| `python -i` prompts at exit only | P1.8: `WaitForMultipleObjects` on pipe reads does not wake on data | Blocking `ReadFile` on stdout; `PeekNamedPipe`+read for stderr on reader thread |
| `>>> >>> >>>` on one line | Several `>>> ` chunks batched in one `processStreamAfterRead` | Process each queue chunk immediately; `ProcessSlashR` on partial flush |

**Mandatory manual regression** after every Phase 1 code change (before marking progress `done`): see §5 Phase 1 — Verification gate.

## 5. Phased implementation plan

### Phase 0 — Baseline and tests (no behavior change)

1. Inventory all `CChildProcess` / `WriteInput` callers (script engine, executor, `proc_input`, plugin API).
2. Repro matrix (see [CChildProcess-repro-matrix-P0.2.md](CChildProcess-repro-matrix-P0.2.md)):

### Phase 1 — Pipe I/O reliability (fixes stuck output)

**Contract:** all steps must comply with **§2.8 Implementation invariants**. A step is not complete without the **Verification gate** below.

**Stdout (read path)**

| Step | Intent | Invariant checkpoints |
|------|--------|------------------------|
| P1.1 | Extract `CChildProcessIo`; `Create()` delegates | Pipe creation unchanged; `onChildProcessStarted()` closes inherited child ends |
| P1.2 | Reader thread → queue; script thread prints | No `PrintOutput` on reader; script thread only |
| P1.3 | No `PeekNamedPipe` on reader | Blocking read **only** on reader thread |
| P1.4 | Exit drain | Kill before drain on break; drain **only** if `!STILL_ACTIVE`; no script-thread blocking read while child runs |
| P1.5 | Less `Sleep` / decouple wait | Must not reintroduce script-thread blocking read or drain-while-alive |
| P1.6–P1.8 | Stdin queue, flush policy, stderr | Must not break §2.8 exit ordering or handle lifecycle |

**Stdin (`WriteInput` — first-class in this phase)**

6. **Queued stdin writes:** UI/executor enqueues; writer thread calls `WriteFile` (avoid blocking UI on `FlushFileBuffers`).
7. Avoid default `FlushFileBuffers` unless explicitly requested (hang risk documented in `WriteInput`).

**Optional**

8. Separate stderr pipe → `PrintOutput` with error styling.

**Files (expected):** `ChildProcess.cpp/h`, `ChildProcessIo.cpp/h`; touch `NppExecScriptEngine.cpp` only if wait points move.

### Phase 2 — Decouple lifecycle from synchronous `Create()`

1. API split:
   - `Start(commandLine)` — process + reader thread (+ writer thread if Phase 1).
   - `WaitForExit(timeout)` / `IsRunning()` — `CScriptEngine::Do` keeps “block until exit” as default.
   - `Stop()` / existing `Kill()`.
2. Move poll loop out of `Create()` body.
3. Preserve `m_execState.pChildProcess` and `GetRunningChildProcess()` for entire run.
4. **`WriteInput` / `WriteChildProcessInput*` unchanged at public boundary** — valid while stdin pipe open.

Enables: PseudoConsole resize while running; cleaner collateral-script interaction.

---

### Phase 3 — PseudoConsole as first-class transport

1. Default policy: PseudoConsole on when API available (keep `npe_console u-` opt-out).
2. **Dynamic geometry:** RichEdit/font metrics → `COORD`; call `ResizePseudoConsole` on resize/font change.
3. Replace fixed `192×1000` with defaults + caps.
4. **P3.5** Review **global** `CreatePseudoConsole` flags (default `0`; document optional API flags only if justified for all children — **not** per-app).
5. UTF-8 / encoding alignment with `getPseudoConsoleEncoding()`.
6. ~~Auto-select PseudoCon vs pipes by executable type~~ — **rejected** (§1.1); transport is user option + global capability only.

**Known note (2026-05-27):** In `u+` (ConPTY), interactive input may appear duplicated
(typed line retained in NppExec console + child-side terminal echo). This was observed
with `gawk getline` and `python -i`. Decide policy in Phase 5: keep terminal echo and
suppress local echo, or preserve local echo and suppress child echo where possible.

**Known note (2026-05-27):** `sfk.exe` (Swiss File Knife) reports a non-terminal style
environment under `u+` (ConPTY) and shows its fallback/install prompt text; under `u-`
(pipe mode) it behaves like a normal terminal run. Treat as app-side ConPTY compatibility
variation; keep generic transport policy (no app-specific code paths).

---

### Phase 4 — VT parser + RichEdit rendering (Scope B)

1. New `CConsoleVtParser` (or extend `CNppExecConsole`): byte-stream CSI/SGR/cursor/erase.
2. Minimum VT: SGR colors, EL/ED, cursor up/down/left/right, `\r` without `\n`.
3. Map to RichEdit: `SetCharFormat`, `ReplaceSelText`, logical cursor in **output region** (above locked input).
4. Implement real **`escProcess`**; PseudoConsole default = process, not strip.
5. Thin `readPipesAndOutput`: dequeue bytes → parser → console ops; keep NppExec filters if needed on logical lines.
6. Document limits: monospaced font, grid approximation, alternate screen optional/deferred.

#### Phase 4.6 — RichEdit/grid limitations (documented behavior as of 2026-05-27)

Current `e2`/`escProcess` behavior is intentionally **terminal-like but not a full terminal emulator**.

| Area | Current behavior | Limitation / impact |
|------|------------------|---------------------|
| VT transport | Stateful parser over output byte stream (`read -> queue -> process`) | Parser is not a full xterm/Windows Terminal state machine. |
| SGR colors | Basic foreground SGR (`30-37`, `90-97`, reset `0`/`39`) | No full attribute palette yet (background, bold/faint, underline, inverse, 256/truecolor). |
| Erase ops | `CSI K` / `CSI J` mapped to output-region erase helpers | Semantics are approximated for RichEdit; not a full screen buffer model. |
| Cursor ops | `\r` and `CSI n D` (left) supported through existing overwrite/backspace primitives | No complete 2D cursor addressing (`H`, `f`, absolute row/col), scroll regions, or DEC modes yet. |
| Output region protection | VT erase/backspace constrained to child output region; internal NppExec status lines stay sticky | RichEdit remains a single text document, not an isolated terminal viewport buffer. |
| Input model | Cooked line input (`WriteInput` + newline) still default | Raw key-by-key terminal input behavior deferred to Phase 5. |
| ConPTY app compatibility | Most CLIs improve under `u+`; fallback `u-` remains available | Some apps (e.g. `sfk.exe`) still detect ConPTY as non-classic terminal and alter UX. |
| TUI fidelity | Basic CLI improvements in-pane | Full-screen TUIs/curses fidelity remains Scope C / optional track, likely external terminal or embedded WT. |

Operational guidance:

- Prefer monospace fonts in Console for best grid approximation.
- Use `npe_console e2` for VT processing; `e1` remains strip-only fallback.
- Keep `u-` as a generic compatibility fallback when a tool behaves poorly under ConPTY `u+`.

**Known note (2026-05-27):** in interactive `cmd` under `u+`, `cls` does not visually
clear the NppExec console. Treat this as expected until VT/screen-buffer semantics are
implemented in Phase 4 (and reconciled with input-echo policy tracked for Phase 5).

### Phase 5 — Input model enhancements (extends existing `WriteInput`)

**Not** “add input channel” — **extend** current path:

| Mode | Behavior |
|------|----------|
| **Cooked (default)** | Current: `ExecuteChildProcessCommand` → `WriteInput(text)` + `WriteInput(GetNewLine())`; `LockConsoleEndPosAfterEnterPressed`. |
| **Raw (optional)** | `RichEditWndProc`: key down → bytes (arrows, Tab, Ctrl+C `\x03`) → `WriteInput` without forced newline. |
| **Signals** | Ctrl+C / Ctrl+Break via executor; ConPTY-aware where applicable. |
| **Scripts** | `proc_input`, `proc_signal` — same encoding/newline rules as console. |
| **Resize** | ConPTY resize (Phase 3) as WINCH equivalent for size-sensitive apps. |

## 6. Implementation order

```
Phase 0 (tests)
    ↓
Phase 1 (async read + queued WriteInput; no PeekNamedPipe)
    ↓
Phase 2 (Start / Wait API)
    ↓
Phase 3 (PseudoCon + ResizePseudoConsole)
    ↓
Phase 4 (VT parser → RichEdit)
    ↓
Phase 5 (raw input, signals on WriteInput path)
    ↓
Phase 6 (optional external / embedded terminal)
```

Phases **1–2** improve reliability without visible ANSI. **3** helps many CLIs; **4** is the main “looks like a terminal” step inside the existing pane.

## 8. Non-goals / constraints

- RichEdit 2.x is not xterm.js; perfect TUIs (Scope C) may be impossible in-pane.
- Windows-centric (PseudoConsole / ConPTY).
- Blocking `CScriptEngine::Do` is entrenched; async child needs explicit product decision.
